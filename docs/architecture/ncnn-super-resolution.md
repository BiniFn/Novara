# NCNN Vulkan Super-Resolution Integration Guide

This document captures the architectural insights, root cause analyses, and solutions derived during the integration of native NCNN GPU super-resolution engines (RealCUGAN and RealESRGAN) into the Kototoro Android application.

## Overview
The goal of this architectural component is to provide real-time, hardware-accelerated AI image upscaling (e.g., 200% enhancement) for manga image tiles directly on the user's mobile GPU, eliminating network latency and preserving visual quality for low-resolution upstream sources. The integration utilizes Tencent's `ncnn` framework with Vulkan compute backend.

## 1. Vulkan Shader GLSL Strictness & `SIGSEGV` Traps

### The Issue
During the initial integration, whenever the pipeline initialized (specifically at `ncnn::Pipeline::create`), the application would crash instantly with a native `SIGSEGV` fault on the Android dispatcher thread.

### Root Cause
The root cause was Khronos `glslang` compiler's rigid syntax enforcement in the latest NCNN Vulkan implementations:
1. **Implicit Type Casting**: Operations like `1 / 255.f` (integer divided by float) are strictly illegal in `vk_glslang`. Standard desktop drivers might forgive them, but mobile driver compilers will crash the pipeline.
2. **Multiple #version Declarations**: NCNN automatically prepends `#version 450` to the shader payload when constructing the SPIR-V blobs. If the actual `.comp` glsl file also starts with `#version 450`, the double declaration creates a syntax fault.
3. **White-space Padding**: Trailing newlines and invisible carriage returns in `.comp` files disrupt the NCNN hex-dump ingestion mechanism.

### The Solution: Automated Shader Sanitization Pipeline
We implemented an automated Python sanitizer (`fix_shaders.py`) inside `app/src/main/cpp/` that executes during the CMake build phase. This script:
- Parses all raw `.comp` files.
- Strips any existing `#version XXX` tags.
- Identifies and rewrites illegal casts (e.g., `1 / 255.f` -> `1.f / 255.f`).
- Compiles the cleaned strings into strictly formatted C-style `.hex.h` headers.

## 2. Uninitialized GPU Calculation Scopes (`VK_ERROR_DEVICE_LOST` / `-4`)

### The Issue
After successful shader compilation, the GPU would immediately lose synchronization and drop the device connection (Vulkan error `-4`), occasionally bringing the entire Android UI composition engine down with it.

### Root Cause
In `realcugan.cpp` and `realesrgan.cpp`, critical variables governing the chunking mathematics were completely implicitly defined (lacking constructor default values):
- `tilesize`: Defines the hardware compute grid patch size (e.g., 400x400).
- `scale`: Super-resolution upscaling multiplier (e.g., 2x, 4x).
- `prepadding`: Safe edge margins to prevent overlap seams.

When called via our Custom JNI bridge `initNative`, these member variables remained uninitialized (taking on OS junk memory values like `-134102940`). This caused the chunking loop `(width + tilesize - 1) / tilesize` to either allocate massive Terabyte-sized single matrices or execute infinitely, instantly triggering hardware watchdog resets.

### The Solution: Rigorous GPU State Initialization
We enforced hardware-appropriate initialization profiles directly within the JNI instantiation:
```cpp
// RealCUGAN Defaults
realcugan->scale = 2;
realcugan->tilesize = 400; // Optimal sweet-spot for Snapdragon Adreno/Mali
realcugan->prepadding = 3;
realcugan->syncgap = 3;

// RealESRGAN Defaults
realesrgan->scale = 4;
realesrgan->tilesize = 400;
realesrgan->prepadding = 10;
```

## 3. The "Android Bitmap Stride" Memory Misalignment

### The Issue
Even after calculations successfully executed, the output generated heavily garbled, diagonally skewed matrices with yellow bars, looking like a shredded TV broadcast.

### Root Cause
We originally attempted "Zero-Copy" processing by directly passing the pointers generated by `AndroidBitmap_lockPixels` to NCNN's `Mat` constructor:
```cpp
ncnn::Mat outMat(outInfo.width, outInfo.height, outPixels, (size_t)4u, 4);
```
However, modern Android system UI pipelines (surface buffers) apply heavy byte-padding optimizations to rows to hit CPU 64-byte or 128-byte cache lines. 
- For instance, a 1272-pixel width image (5088 bytes) might be aligned to a `stride` of 5120 bytes.
- NCNN computational math fundamentally assumes that `RowSize == Width * Channels` without exception. 
- Because NCNN operated under the continuous matrix assumption, every single Y offset generated an expanding 32-byte physical drift into the `outPixels` rendering buffer, permanently shifting all pixel channels into diagonal misalignments.

### The Solution: "Smart Dense Packing" Bridge
We replaced the risky Zero-Copy implementation with a dynamically resilient memory marshaller in the JNI:
- The system interrogates the Android target buffer: `bool needsCopy = (info.stride != info.width * 4);`
- If padding is detected, we allocate a densely packed heap intermediate matrix.
- We perform a high-speed pointer-jumping `memcpy` loop to copy Android's padded format into the dense contiguous array.
- NCNN performs flawless mathematics on the dense array.
- Post-processing, the array is decompressed back into the padded Android stride geometry.

```cpp
if (outNeedsCopy && ret == 0) {
    for (uint32_t y = 0; y < outInfo.height; y++) {
        memcpy((uint8_t*)outPixels + y * outInfo.stride, 
               denseOut + y * outInfo.width * 4, 
               outInfo.width * 4);
    }
}
```

This ensures absolute calculation stability regardless of whether the specific user's SoC OEM modifies the OS-level bitmap graphics pipeline buffer alignment rules, and completely eliminates the visual tearing artifacts.
