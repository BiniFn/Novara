# Hero 转场性能改进计划

## 背景

当前应用的 hero 动画基于 Jetpack Compose `SharedTransitionLayout` + `sharedElement()` 实现，功能上可用，但在 Discover / Explore / 列表页进入详情页时存在明显卡顿。

现状问题不是单点故障，而是以下因素叠加：

- 来源页 hero 本身存在持续背景动画、`Crossfade`、实时 blur 等高成本效果
- 详情页首屏在转场开始时同步进入大量状态收集、复杂 header 计算和图片请求创建
- 来源页与目标页的封面图片请求没有统一缓存键，转场期间更容易出现重新解码或纹理切换
- 共享元素当前包裹的是带阴影、边框、背景和 padding 的容器，而不是尽量纯净的图片层

## 目标

### 用户目标

- 点击封面进入详情页时，hero 动画明显更顺滑
- 动画期间减少封面闪动、跳变、重载感
- 返回时保持现有功能正确性，不引入封面错位或页面空白

### 工程目标

- 优先以低风险方式减少首屏帧预算压力
- 不引入新依赖
- 不重构导航和业务状态流
- 先消除最明显的性能浪费，再决定是否进入第二阶段结构重构

## 问题拆解

### 1. 来源页渲染负担过重

`DiscoverHeroCarousel` 当前在 hero 区域同时承担：

- 无限背景缩放和平移动画
- 背景图片切换 `Crossfade`
- 可选实时 `blur`
- 前景封面和文案

这意味着用户点击封面触发导航时，来源页本身就已经在持续消耗 UI / RenderThread / GPU 预算。

### 2. 详情页首屏过重

`DetailsScreen` 在首次进入时会立刻开始：

- 多个 `collectAsStateWithLifecycle()`
- 复杂 header 数据准备
- 文件大小 `computeSize()`
- 额外图片请求和文案列表构建

这部分工作与 hero 动画竞争同一段关键帧时间窗口。

### 3. 图片缓存策略不一致

详情页封面已经有稳定 `memoryCacheKey` / `diskCacheKey`，但来源页 hero / explore / 列表封面未统一复用相同缓存键。

结果：

- 共享元素期间更容易发生重新加载
- 即使图片 URL 相同，也未必能稳定复用同一个内存纹理
- 动画时更容易出现切换感或轻微掉帧

### 4. 共享元素包裹范围过大

当前 `sharedElement()` 附着在详情页封面外层容器上，容器还叠加了：

- `shadow`
- `background`
- `border`
- `padding`

这会扩大共享元素的测量和绘制范围，也增加视觉不一致风险。

## 设计原则

### KISS

优先做：

- 统一缓存键
- 转场窗口内降载
- 缩小共享元素范围

不优先做：

- 大规模拆分 `DetailsScreen`
- 重写导航过渡体系
- 引入额外动画框架

### YAGNI

第一阶段只做当前明确需要的性能修复，不预留复杂的通用转场框架。

### DRY

- 抽出统一的封面缓存键构建函数
- 抽出共享的 hero 转场状态 `CompositionLocal`

### SOLID

- 转场状态与具体页面解耦
- 缓存键逻辑从详情页私有实现提升为共享工具
- 来源页与目标页只依赖抽象的转场状态，不直接互相引用

## 实施阶段

## 阶段 1：低风险高收益修复

### 1. 建立共享 hero 转场状态

新增 `CompositionLocal<Boolean>`，表示当前是否处于 hero 导航关键窗口。

作用：

- 来源页在点击后立刻降载
- 目标页在进入初期延迟渲染重内容

### 2. 统一封面缓存键

抽出共享工具函数，基于：

- 共享元素 key
- source name
- content owner key
- cover url

在来源页和详情页统一构建 `memoryCacheKey`，并让详情页优先用来源页的缓存图作为过渡基底。

### 3. 缩小共享元素范围

只让真正的封面图片参与 `sharedElement()`。

保留外层容器的阴影、边框、背景，但不将其纳入共享元素计算范围。

### 4. 转场期间降载

在 hero 导航进行中：

- 暂停 Discover hero 无限背景动画
- 禁用 hero 背景 `Crossfade`
- 禁用实时 blur，保留更轻的静态路径
- 详情页延迟显示非关键 header 内容
- 避免转场初期执行本地文件大小计算

## 阶段 2：结构性优化

### 1. 继续拆分详情页首屏

目标是把详情页切成：

- 转场关键内容：封面、标题、返回按钮
- 延迟内容：统计、来源、追踪、说明、补充信息

### 2. 检查 `sharedElement()` 与 `sharedBounds()` 分工

如果未来仍存在容器形变、圆角/边框跳变，可考虑：

- 图片层使用 `sharedElement()`
- 卡片容器使用 `sharedBounds()`

### 3. 根据基准测试结果决定是否继续

只有在第一阶段收益不足时，才继续进入更高成本的结构重构。

## 验证方式

### 功能验证

- Discover hero 进入详情页正常
- Explore 列表进入详情页正常
- 返回动画无明显错位
- 封面加载失败 fallback 逻辑保持正常

### 性能验证

- `Layout Inspector` 观察转场期间重组热点
- `Perfetto` 或 `System Trace` 观察导航触发后的掉帧情况
- 建议使用 release 构建验证，而不是 debug

### 重点观察指标

- 点击进入详情页后的前 300ms 帧稳定性
- 封面是否闪动或重新淡入
- 详情页 header 是否在动画开始瞬间大面积重排

## 风险与回滚

### 风险

- 详情页延迟内容渲染可能导致短暂的信息后出现
- 缓存键统一后，如果 key 设计不稳定，可能影响部分封面缓存命中
- 缩小共享元素范围后，视觉效果会从“整卡飞入”变成“封面飞入”，这是有意的取舍

### 回滚策略

- hero 转场降载逻辑可单独移除
- 缓存键共享工具可保留，不影响功能正确性
- 共享元素附着范围可回退到旧容器实现

## 本次实施范围

本次按计划实施阶段 1：

- 新增共享 hero 转场状态
- 统一共享元素封面缓存键
- 缩小共享元素范围到封面图片层
- 来源页转场期间降载
- 详情页转场期间延迟重内容

阶段 2 暂不实施，等待第一阶段结果验证后再决定。

## 相关后续修复

在 hero 转场优化过程中，排查出一个与跨设备恢复相关的数据一致性问题：

- metadata source 选择与显示 override 存在于 `preferences` 表
- 旧备份只导出了 `MangaEntity`
- 恢复后列表页缺少 metadata source 选择，可能显示原始封面
- 详情页因运行时重新解析或局部数据存在，可能又显示替换封面，形成不一致

已修复方向：

- `ContentBackup` 增加 `title_override` / `cover_override`
- `ContentBackup` 增加 `metadata_source_kind` / `metadata_source_service` / `metadata_source_remote_id`
- restore 时同步回写 `preferences` 表对应字段

这部分不直接提升 hero 性能，但能避免跨设备恢复后因为 metadata source 丢失导致的封面与详情页状态错乱。
