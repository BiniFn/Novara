# Kototoro Compose 迁移洞察

## 对比 `task.md` 进度结论

| Phase | 文档声明 | 实测 | 说明 |
|-------|---------|------|------|
| 1 | 进行中 | ✅ 部分落地 | 设置渲染层全部 Compose；搜索索引/DSL warning/tabbed host 仍为 XML |
| 2 | 待启动 | ⚠️ 部分启动 | `DownloadDialog` / `ContentStatsDialog` / `ScrobblingSelectorDialog` 新 Compose 版已就位但 **L1 壳仍在**；`ChaptersPagesSheet` 仍是 XML+Compose 混合 |
| 3 | 待启动 | ❌ 未启动 | `DetailsActivity` 仍然依赖 `ActivityDetailsBinding` + `supportFragmentManager.findFragmentById` |
| 4 | 待启动 | ❌ 未启动 | `GlassSurface` 仍为 fallback Surface；无 GlassCard / GlassSheet |
| 5 | 远期 | ❌ 未启动 | 仓库无 `shared/` 或 `commonMain/` |

结论：整体 Android 端 Compose 覆盖率 ~70%，但存在多处 **已迁移但旧路径未清理** 的重影代码（死代码、无效副作用、重复状态），直接拉低了"完善度"和可读性。

## 一、高优先级 Bug（直接影响功能）

### 1.1 `AppContentListRoute` 禁用了无限加载
**文件**：`list/ui/compose/AppContentListRoute.kt:107`
```kotlin
onLoadMore = { },
```
**影响**：通过该 route 承载的 **收藏夹列表**、**Local、Suggestions、Updates、Search** 等所有页面都无法触发分页加载，用户滚动到底不会拉到下一页。`FavouritesListViewModel.requestMoreItems()` 存在但被隔离。  
**修复**：通过新参数或 `viewModel::requestMoreItems` 反射式探测来接回 `onLoadMore`。

### 1.2 Favorites 页面顶部双重 padding
**文件**：`favourites/ui/compose/FavoritesHostScreen.kt:68`、`list/ui/compose/AppContentListRoute.kt → KototoroContentListScreen`
- 外层 `Column.padding(top=contentPadding.top)` 把 tab row 压到 top bar 下方。
- 内层 `LazyVerticalGrid.contentPadding = contentPadding`（仍带 top 值）再次把内容往下推 top 一次。
**影响**：收藏夹标签下第一排卡片多出约 `topBarHeight` 的空白。  
**修复**：在 `HorizontalPager` 内传入 `contentPadding = PaddingValues(bottom, start, end)`（剥离 top）。

### 1.3 `DetailsAction.OpenTracking` / `OpenStatistics` 在 Activity 侧是死代码
**文件**：`details/ui/DetailsActivity.kt:260-270`
```kotlin
DetailsAction.OpenTracking -> {
    viewModel.getContentOrNull()?.let { m ->
        intent.putExtra(AppRouter.KEY_MANGA, arrayOf(ParcelableContent(m)))
    }
}
```
对当前 intent `putExtra` 没有任何 UI 效果。真正的对话框是在 `DetailsScreen.kt` 内用 `showScrobblingDialog` / `showStatsDialog` state 驱动 Compose `AlertDialog`。
**修复**：去掉 Activity 侧对这两个 action 的处理分支；`DetailsScreen` overflow menu 调用链也无需再二次转发给 `onActionClick`。

### 1.4 HomeScreen 有 ~1500 行死代码
**文件**：`home/ui/compose/HomeScreen.kt`
当前可见渲染只使用 `HomeContentCarouselCard` × 3、`HomeTrackingHeroSection`、`QuickActionsCard`。
但文件内仍保留 `HomeHeroSection` / `HomeUtilityHubCard` / `HomePulseCard` / `HomeCollectionSection` / `UtilitySection` / `TrackingSpotlightRow` / `ResumePanel` / `OverviewMetricCard` / `HomeHeroLinkRow` / `HomeActionChip` / `HeroFeatureArtwork` / `HeroPoster` / `HeroThumbnail` / `ContentLazyRow` / `ContentColumn` / `ContentListItem` / `ContentCoverItem`、`HomeHeroTab`/`HomeHeroTabSpec`、`UtilityHeader`、`HomeHeroDetails`、`HomeHeroActionRow`、`HomeHeroActionButton`、`HomeHeroTabRow`、`HomeTrackingSpotlight` 等。
**影响**：
- 增加维护负担，KSP/编译时间；
- 未来 CMP 拆分时被迫整体搬运。
**修复**：删除未引用的私有组件；保留 `HomeBadge` / `HomeStatPill` 等仍被使用的辅助。

### 1.5 `DetailsActivity.onCreate` 启动两条 "postponed transition" 路径
**文件**：`details/ui/DetailsActivity.kt:124-127` 与 `308-317`
```kotlin
supportPostponeEnterTransition()
window.decorView.postDelayed({ supportStartPostponedEnterTransition() }, 350)
...
viewBinding.imageViewCover.addImageRequestListener(object : coil3.request.ImageRequest.Listener { ... supportStartPostponedEnterTransition() ... })
```
**影响**：首次打开时，350ms 定时器可能在图片加载完成前强行开始转场，导致封面"闪黑/闪回"。  
**修复**：取消定时器路径，仅依赖图片加载完成回调；或将超时保护提到更长（例如 1200ms）仅作为 fallback。

### 1.6 `DiscoverViewModel` 有生产日志 & 双重重入保护缺失
**文件**：`discover/ui/DiscoverViewModel.kt:226`
```kotlin
android.util.Log.d("DiscoverPagination", "loadData initiated: ...")
```
- 生产版本保留 `Log.d` 不符合项目风格；
- `loadNextPage` 的 `isPageLoading` 不是原子的，高频快速滚动可能让多个页面同时加载。  
**修复**：去掉 debug 日志；把 `isPageLoading` 升格为 `AtomicBoolean` 或放在 `withContext` 下用 `Mutex`。

### 1.7 `KototoroTopBar` "TopBarAnchorIconButton" 使用 AndroidView 嵌入不可见 View
**文件**：`main/ui/compose/KototoroTopBar.kt:242`  
为获取 anchor View 给 Popup 用，塞了一个 1×1 的 `AndroidView`，与 Compose 的 `LocalView` 不一致，还占 measure slot。  
**修复**：直接 `val anchor = LocalView.current`，摒弃隐藏 View。

### 1.8 `AppNavGraph` Home 跳转 History 时不走 NavHost
**文件**：`main/ui/compose/AppNavGraph.kt:52`
```kotlin
onViewAllRecentClick = { appRouter.openHistory(BrowseGroupTab.Content) }
```
走 Activity 路径而非 `navController.navigate("history")`，造成双页栈、失去 Compose 转场连续性。  
**修复**：`navController.navigate("history")`。同理 `onLocalClick` 也应用 navController。

### 1.9 FavouritesContainerViewModel 印出 `println` 日志
**文件**：`favourites/ui/container/FavouritesContainerViewModel.kt:98-99`
`logImport/logSync` 直接写 stdout。Release 包里会无序输出。  
**修复**：统一替换为 `Timber`/项目已有日志 wrapper，或 guarded by `BuildConfig.DEBUG`。

## 二、UI/UX 设计不足

### 2.1 Home 轮播卡片信息密度低，缺少评分/状态
三张 `HomeContentCarouselCard`（历史/更新/推荐）只展示标题 + 来源。与 Tracking Hero Section 相比缺少评分、进度、上次阅读章节等维度信息，无法形成视觉对比。

### 2.2 Tracking Hero 高度硬编码 184dp
`HomeTrackingHeroSection` 与 `TrackingCategoryCarouselCard` 都写死 `.height(184.dp)` + 内部 `.height(100.dp)` poster。当字体放大/内容过长时会挤压。  
**建议**：改为 `minHeight` + 内容自适应。

### 2.3 Explore 顶部"Sources"卡使用通用 `ic_storage`
每个 source 无实际 logo，辨识度低。  
**建议**：若 source 提供 favicon/domain，使用 `AsyncImage` + fallback。

### 2.4 空态文案硬编码英文
`KototoroContentListScreen.kt:63`
```kotlin
Text(text = "No content available", ...)
```
应该使用资源。

### 2.5 FavoritesHostScreen 空态未考虑平板/大屏
Icon 固定 64dp + 居中；在 foldable 外屏很小；大屏又显空荡。  
**建议**：大屏下提供 "添加收藏夹"/"浏览源" CTA。

### 2.6 Details 底部 Read Dock 文本在窄屏下溢出
`ReadDock` 主按钮文案与 branch 名并列但未做权重控制，窄屏 + branch 名长时 KebabArrow 可能被顶出。

### 2.7 GlassSurface 标注 "Temporary fallback" 未落地
`GlassSurface.kt:80` 注释里说"while haze-backed glass is not wired through the main shell"。当前所有 Glass* 都只是半透明 Surface。整个"液态玻璃"视觉语言目前只是命名空间。  
**建议**：接入 `dev.chrisbanes.haze` 的 `HazeChild` + `HazeState`，或渐进降级到 `BlurProvider` 抽象。

### 2.8 顶栏搜索框展开后内容被底部导航栏遮挡
`DockedSearchBar` 展开后建议列表填满屏幕，但 `navigationBarsPadding` 没有应用到 suggestion LazyColumn，底部底项有可能被主壳 bottom nav 遮挡。

### 2.9 ChaptersPagesSheet 仍用 XML TabLayout + Material Sheet
内部分页已 Compose，外壳还是 `BaseAdaptiveSheet + TabLayout`。过渡动画与 Compose 端不统一，且 `TabLayout.getTabAt`/`selectTab` 与 Compose `PagerState` 之间存在竞争（两端互相 sync 时都要 `composeScope.launch`）。

### 2.10 Discover Hero Carousel 与 Tracking Category Carousel 视觉风格重复
两者都是 BackdropCard + HorizontalPager + 封面 blur + 评分徽章。Home、Explore、Discover 都在复制相似代码，应抽成通用 `ContentHeroBackdropCarousel`。

## 三、架构与代码质量

### 3.1 `MainActivity` 仍有大量空/无用函数
```kotlin
fun selectMainNavigationItem(...): Boolean { return false }
override fun addMenuProvider(...) { super.addMenuProvider(...) }  // 透传
onSupportActionModeStarted / Finished / setNavbarPinned / setNavFloating / setNavHeight ...
```
这些是上一版 `MainNavigationDelegate` 移除后遗留的。

### 3.2 `MainActivity` 大量 `mutableStateOf` 字段
在 Activity 字段上保存 Compose state 是可行的，但更 idiomatic 的做法是把它们下沉到 `MainViewModel.StateFlow`。目前的做法导致 `MainActivity` 变成 500+ 行半状态容器。

### 3.3 AppNavGraph 里嵌套 hiltViewModel + MenuProvider + DisposableEffect
每个 composable destination 都内联 200~300 行 ViewModel 注入、事件收集、菜单注册。建议每个路由抽到独立的 `AppXxxRoute` 文件（已经有 `AppContentListRoute` 前例），NavGraph 只负责连线。

### 3.4 `AppContentListRoute.DisposableEffect(mainActivity, viewModel)` 在 Detail Activity 下会 NPE 风险
`LocalContext.current as? androidx.activity.ComponentActivity` + `as? MainActivity`。如果被放到 DetailsActivity 或其他 Activity 再 reuse，filter callback 就消失而且不会报错 —— 这种 silent no-op 调试困难。

### 3.5 `KototoroApp` 构造参数膨胀（30+ 个回调/state）
一个 Composable 接收 30+ 参数是明显坏味。建议抽为 `MainAppState data class` 或者 `@Stable class MainAppController`。

### 3.6 `DetailsScreen` 生命周期 snapshot 用 `collectAsState` 而非 `collectAsStateWithLifecycle`
会在后台仍然订阅，浪费协程。改用 `collectAsStateWithLifecycle`。

### 3.7 `DownloadDialog` 存在重复实现
- `download/ui/compose/DownloadDialog.kt`（新 L2 版）
- `download/ui/dialog/compose/DownloadDialog.kt`（L1 via DialogFragment 壳）
两份实现同名不同包，维护心智成本高，**git status**/`grep` 经常误导开发者。

### 3.8 `ReadButtonDelegate` 在 Compose 里仍被 Sheet 层 Attach
Compose 端 `DetailsScreen` 有自己的 `ReadDock`（Compose），而 `ChaptersPagesSheet` 里仍用 `ReadButtonDelegate(binding.splitButtonRead, ...)` attach 一份，两个按钮做同一件事，状态同步靠 ViewModel。L2 后应下线其中一个。

### 3.9 主 Activity 仍用 `ActivityMainBinding` 加载
`MainActivity:172` 
```kotlin
setContentViewWebViewSafe { ActivityMainBinding.inflate(layoutInflater) }
```
虽然 `activity_main.xml` 已精简为单个 ComposeView，但 `setContentViewWebViewSafe`（继承自 BaseActivity）要求 ViewBinding。这是"Compose 主壳但 WebView 救急 shim"的残留。  
**建议**：重构 `BaseActivity` 暴露 `setContent { ... }` 直接 Compose 入口，或在 MainActivity override。

### 3.10 FragmentHostRoute 已 archiv（好事）
`FragmentHostRoute.kt` 已删除（diff 显示 `-31`）。保持。

## 四、多平台潜力（Phase 5 前置）

1. 现有 `core/ui/compose/*` / `core/ui/glass/*` / 通用 Hero 组件已天然靠近 commonMain，仅绑定 `androidx.compose.material3` 与 `coil3`。
2. 阻碍 commonMain 的主要依赖：
   - `androidx.fragment.app.*`（Favorites / ChaptersSheet / AppContentListRoute 的 `ComponentActivity`）
   - `androidx.hilt.navigation.compose.hiltViewModel`
   - `android.view.View` 作为 anchor 被多处 route 传递（Popup menu、filter chip、source tag dropdown）
   - `DateUtils`、`WindowInsetsCompat`
   - `coil3.request.ImageRequest.Builder(Context)`
3. 路线图应是：先把"组件 → Android 平台 anchor"的耦合变成 expect/actual，再拆 `designsystem` 模块。
